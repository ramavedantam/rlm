# Romantic Language Timeline: 50-Novel RLM Experiment Analysis

**Run:** 2026-04-09 13:30:46 UTC
**Model:** gpt-5.4 (OpenAI)
**Corpus:** 50 novels, 10 decades (1800s-1890s), 43MB total text
**Cost:** 992,210 tokens (940K input, 51K output)
**Iterations:** 11 (all at depth 0)

## What This Experiment Tests

The Recursive Language Model (RLM) gives an LLM a Python REPL with persistent state and two special functions: `llm_query()` (single-shot LLM call) and `rlm_query()` (spawn a full child RLM with its own REPL). The model writes code, sees output, iterates.

This experiment asks: given 50 novels across a century, can the model invent a computational pipeline to analyze romantic language evolution -- without being told how?

The query asks for per-novel metrics (frequency per 10,000 words, sentiment intensity 1-5, dominant themes), decade aggregates, a century timeline with inflection points, and the top 5 most romantically intense passages with exact quotes. The corpus is too large for a single LLM call, the metrics require real computation, and exact quotes require grounding in source text.

## What the Model Did

The model produced 11 code blocks totaling ~330 lines of Python. No human guidance, no scaffolding. It invented a three-stage pipeline:

**Stage 1: Code-based extraction** (blocks 1-6). Explore the directory, read all 50 novels, strip Gutenberg boilerplate, split into paragraphs, filter by romance keywords.

**Stage 2: LLM-based classification** (blocks 7-8). Score candidate passages with a heuristic, keep the top 80 per novel, batch 10 novels at a time into `llm_query` for romantic classification, sentiment scoring, and theme tagging.

**Stage 3: Code-based aggregation** (blocks 9-11). Parse all LLM outputs, compute per-decade averages, calculate decade-over-decade deltas, rank top passages corpus-wide, assemble final JSON.

The key insight: the model decided **what needs code** (keyword matching, frequency math, averaging) vs. **what needs judgment** (is this passage genuinely romantic? how intense is it?). The LLM is a subroutine called where judgment is needed; everything else runs as Python.

## The Code

### Block 6: Gutenberg Stripping + Romance Keyword Extraction (57 lines)

The model invented a 60-keyword romance vocabulary and a compiled regex to scan all 50 novels at paragraph level:

```python
def strip_gutenberg(text):
    start_markers = [
        "*** START OF THE PROJECT GUTENBERG EBOOK",
        "*** START OF THE PROJECT GUTENBERG",
    ]
    end_markers = [
        "*** END OF THE PROJECT GUTENBERG EBOOK",
        "*** END OF THE PROJECT GUTENBERG",
    ]
    start = 0
    for m in start_markers:
        idx = text.find(m)
        if idx != -1:
            nl = text.find("\n", idx)
            start = nl + 1 if nl != -1 else idx
            break
    end = len(text)
    for m in end_markers:
        idx = text.find(m)
        if idx != -1:
            end = idx
            break
    return text[start:end].strip()

clean_texts = {k: strip_gutenberg(v) for k, v in raw_texts.items()}

romance_keywords = [
    "love", "loved", "loving", "beloved", "adore", "adored", "adoration",
    "passion", "heart", "affection", "tender", "tenderness", "dear", "darling",
    "marry", "married", "marriage", "proposal", "propose", "engaged", "engagement",
    "kiss", "embrace", "embraced", "court", "courtship", "suitor", "woo", "wedded",
    "jealous", "jealousy", "devotion", "attachment", "admire", "admiration",
    "beauty", "beautiful", "charm", "charming", "attraction", "desire", "longing",
    "yearn", "yearning", "fond", "fondness", "lover", "lovers", "mistress", "bride",
    "bridegroom", "wedding", "union", "faithful", "constancy", "constancy", "flame"
]

kw_re = re.compile(
    r'\b(?:' + '|'.join(
        re.escape(k) for k in sorted(set(romance_keywords), key=len, reverse=True)
    ) + r')\b', re.I
)

def split_passages(text):
    paras = [p.strip() for p in re.split(r'\n\s*\n+', text) if p.strip()]
    return paras

candidate_passages = {}
for key, text in clean_texts.items():
    paras = split_passages(text)
    hits = [p for p in paras if kw_re.search(p)]
    candidate_passages[key] = hits
```

**Output:** 923 candidates from Belinda, 879 from Udolpho, 567 from Pride and Prejudice, 119 from Castle Rackrent. Average 452 candidates per novel across the corpus.

### Block 7: Heuristic Scoring + LLM Batch Classification (53 lines)

The model ranked candidates by keyword density and batched the top ones to the LLM for judgment:

```python
def passage_score_heuristic(p):
    text = p.lower()
    score = 0
    score += sum(text.count(k) for k in [
        "love", "marry", "marriage", "kiss", "beloved", "adore",
        "passion", "heart", "affection", "dear", "darling",
        "proposal", "engaged", "wedding", "jealous"
    ])
    if '"' in p or "\u201c" in p:
        score += 2  # dialogue bonus
    score += min(len(p.split()) // 40, 4)  # length bonus
    return score

novel_inputs = {}
for key, passages in candidate_passages.items():
    ranked = sorted(passages, key=passage_score_heuristic, reverse=True)
    top = [normalize_ws(p)[:900] for p in ranked[:80]]
    novel_inputs[key] = top
```

Then batched 10 novels at a time into `llm_query` with a structured prompt requesting JSON output with romantic_passage_count, frequency_per_10k_words, avg_sentiment_intensity, dominant_themes, and top_passages with exact quotes.

### Block 9: Programmatic Aggregation (64 lines)

All aggregation was code, not LLM:

```python
per_decade = {}
for d in decades:
    items = [r for r in novel_results if r["decade"] == d]
    theme_counts = {t: 0 for t in theme_list}
    for r in items:
        for t in r.get("dominant_themes", []):
            if t in theme_counts:
                theme_counts[t] += 1
    dominant = [t for t, c in sorted(theme_counts.items(),
                key=lambda x: (-x[1], x[0])) if c > 0][:5]
    per_decade[d] = {
        "decade": d,
        "novel_count": len(items),
        "avg_romantic_frequency_per_10k_words": round(
            sum(r["romantic_frequency_per_10k_words"] for r in items) / len(items), 3
        ),
        "avg_sentiment_intensity_1_to_5": round(
            sum(r["avg_sentiment_intensity_1_to_5"] for r in items) / len(items), 3
        ),
        "dominant_themes": dominant,
    }

# Decade-over-decade deltas
changes = []
for i in range(1, len(decades)):
    prev = per_decade[decades[i-1]]
    cur = per_decade[decades[i]]
    changes.append({
        "from_decade": decades[i-1],
        "to_decade": decades[i],
        "frequency_change": round(
            cur["avg_romantic_frequency_per_10k_words"]
            - prev["avg_romantic_frequency_per_10k_words"], 3
        ),
        "sentiment_change": round(
            cur["avg_sentiment_intensity_1_to_5"]
            - prev["avg_sentiment_intensity_1_to_5"], 3
        ),
        "theme_shift": [
            t for t in cur["dominant_themes"]
            if t not in prev["dominant_themes"]
        ]
    })
```

### Inflection Points

**1810s to 1820s -- Major contraction.** Romantic frequency drops from 1.82 to 0.89. The 1810s are dominated by Austen (4 of 5 novels); the 1820s shift to Cooper and Scott -- adventure fiction where romance is peripheral. The frequency halves.

**1820s to 1830s -- Resurgence and peak intensity.** Frequency rebounds from 0.89 to 1.89, and intensity hits the century high at 4.3. Jane Eyre (4.4), Wuthering Heights (4.6), and The Hunchback of Notre-Dame (4.5) drive this -- the Gothic/Romantic peak where love is suffering.

**1870s to 1880s -- Tonal collapse.** Frequency holds steady (1.62 to 1.70) but intensity crashes from 3.72 to 2.55 -- the century low. The 1880s corpus includes Jekyll & Hyde (1.0), Treasure Island (1.0), and War of the Worlds (2.67): genre fiction where romance is incidental or absent.

**1880s to 1890s -- Late-century re-intensification.** Frequency rises to 2.14 (century high), intensity rebounds to 3.52. Tess of the d'Urbervilles (4.33), Dracula (4.16), and The Awakening (4.10) bring back romantic suffering and loss -- darker, more psychologically complex.

## Emergent Behaviors

### What the model invented (unprompted)

1. **Romance keyword vocabulary.** 60 terms organized by category (endearments, marriage, physical, emotional). Not a naive wordlist -- includes "constancy," "suitor," "woo," "flame."

2. **Compiled regex with longest-match-first ordering.** `sorted(set(romance_keywords), key=len, reverse=True)` prevents "love" from matching inside "beloved."

3. **Paragraph-level passage splitting.** Not sentence-level (too granular) or chapter-level (too coarse). Split on `\n\s*\n+` -- double newlines. A reasonable unit for romantic scenes.

4. **Multi-signal heuristic scoring.** Keyword density + dialogue bonus (`'"' in p`) + length bonus (`len(p.split()) // 40`). This prioritizes passages where characters speak about love in substantive dialogue.

5. **Top-80 pruning per novel.** 923 keyword-matching paragraphs in Belinda is too many for the LLM. Keep the top 80 by heuristic score, trim to 900 chars each. Manages context window budget.

6. **Controlled theme vocabulary.** Defined 10 theme categories and constrained the LLM to choose from them. This is a software engineering decision about interface design: when the model builds a pipeline where LLM output feeds into code aggregation, it needs the LLM to produce structured, enumerable labels -- not free text. Without this, "forbidden love between social classes," "romantic longing and class-based courtship," and "love across societal boundaries" would all describe the same thing but be uncountable by code. The model solved this by defining an enum up front (`theme_list = ["courtship", "declaration_of_love", ...]`) and instructing the LLM: "dominant_themes = up to 5 themes from this controlled list." This makes the downstream aggregation trivial -- `theme_counts[t] += 1` just works. The model recognized it was designing a data pipeline interface, not just asking a question.

7. **Batched LLM calls with schema enforcement.** 10 novels per batch, 5 batches total. Requested STRICT JSON with named keys. Parsed output with a fault-tolerant JSON extractor that falls back to regex.

8. **Gutenberg header stripping.** Handled both `***START` and `*** START` variants, found newline after start marker.

### What the model chose NOT to do

1. **No recursion.** Despite having `rlm_query()` available, the model handled everything at the root level. It used `llm_query()` as a classification subroutine, not `rlm_query()` for hierarchical dispatch.

2. **No chunking.** The design spec expected the model to chunk novels into smaller pieces for analysis. Instead it used keyword heuristics to extract candidates, bypassing the need to send full text to the LLM.

3. **No per-novel LLM calls.** Instead of 50 individual calls, it batched 10 at a time. 5 total `llm_query` calls for classification + 1 for inflection point analysis = 6 LLM calls total.

## How RLM Differs From Claude Code

This experiment surfaces a fundamental architectural difference between RLM and tool-using agents like Claude Code.

### Claude Code: LLM orchestrates tools

```
LLM thinks -> calls tool -> sees result -> LLM thinks -> calls tool -> ...
```

Every step passes through the LLM. If Claude Code needed to process 50 novels, it would either write one monolithic script via Bash and hope it works, or iterate through 50+ agent loop turns -- each one a full LLM call with the growing conversation context. The LLM IS the program; tools are its I/O.

### RLM: Code orchestrates LLM

```python
for novel in novels:           # code loop, runs at CPU speed
    passages = extract(novel)  # code -- regex, no LLM needed
    scored = llm_query(passages)  # LLM called as a function
    results.append(aggregate(scored))  # code
```

The for loop is in code, not in the agent loop. The LLM is a **subroutine** called only where judgment is needed. Everything else -- keyword extraction, batching, frequency math, averaging, aggregation -- runs as Python at CPU speed.

### Three concrete consequences

**1. Cost.** Claude Code would re-read its full conversation context at every iteration. To process 50 novels with per-novel analysis, that's 50+ full context reads. RLM's code loop runs 50 iterations without touching the LLM, then makes 5 targeted `llm_query` calls. This experiment: 992K tokens. A comparable Claude Code workflow would likely cost 5-10M tokens.

**2. Reliability.** The aggregation math is `sum()/len()` in Python, not "please average these numbers" to an LLM. Decade-over-decade deltas are computed with subtraction, not natural language reasoning. No hallucinated arithmetic. The model chose what needs code vs. what needs judgment -- and made that choice correctly.

**3. Emergent strategy.** Nobody told the model to use keyword heuristics, batch scoring, controlled theme vocabularies, or fault-tolerant JSON parsing. The model invented its own pipeline because it has the freedom to write arbitrary code. In Claude Code, the model can only use the tools it's given. In RLM, the model writes the tools.

### What RLM should enable but didn't (yet)

The design spec expected depth-3 recursion: root dispatches to 10 decade coordinators via `rlm_query`, each dispatches to 5 novel analyzers. This would be the thing tool-using agents truly cannot do: **code that spawns reasoning agents that write more code.**

The model found a cheaper path -- flat execution with `llm_query` batches. Efficient, but it means the recursive dispatch capability of RLM remains untested at scale. Future experiments may need to force recursion by making the task impossible to flatten (e.g., requiring different analysis strategies per decade, or per-novel intermediate results that are too large to aggregate in a single context window).
