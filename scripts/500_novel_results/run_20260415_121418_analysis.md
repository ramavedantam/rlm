# Romantic Language Timeline: 500-Novel RLM Experiment Analysis

**Run:** 2026-04-15 12:14:18 UTC
**Model:** gpt-5.4 (OpenAI)
**Corpus:** 419 novels, 10 decades (1800s-1890s), 309MB total text
**Cost:** 93,527 tokens (87.7K input, 5.8K output)
**Iterations:** 11 (all at depth 0)
**Code:** 485 lines of emergent Python across 11 blocks

## What This Experiment Tests

The Recursive Language Model (RLM) gives an LLM a Python REPL with persistent state and two special functions: `llm_query()` (single-shot LLM call) and `rlm_query()` (spawn a full child RLM with its own REPL). The model writes code, sees output, iterates.

This experiment scales up the 50-novel romantic timeline run by 8x: 419 novels across a century, 309MB of raw text. Can the model build an extraction and analysis pipeline that handles a corpus this large -- without using the LLM for classification at all?

The query asks for per-novel metrics (frequency per 10,000 words, sentiment intensity 1-5, dominant themes), decade aggregates, a century timeline with inflection points, early-vs-late comparison, and the top 10 most romantically intense passages with exact quotes. The corpus is far too large for any LLM context window, the metrics require real computation, and exact quotes require grounding in source text.

The corpus was built programmatically by `build_500_corpus.py`, which queries the Gutendex API for popular English fiction and estimates publication decades from author birth/death years. Early decades (1800s-1820s) have fewer novels due to Gutenberg catalog limits; later decades have 50 each.

## What the Model Did

The model produced 11 code blocks totaling 485 lines of Python. No human guidance, no scaffolding. It invented a **fully code-based pipeline** -- unlike the 50-novel run, it used zero `llm_query()` calls. The entire analysis was computational:

**Stage 1: Exploration and indexing** (blocks 1-5). Discover the decade directory structure, list all files per decade, read a sample novel to inspect Gutenberg header format, build a corpus index mapping decades to file lists.

**Stage 2: Text cleaning and metadata extraction** (block 6). Read all 419 novels, strip Gutenberg boilerplate (handling three header format variants), extract title/author from metadata fields with regex fallback to filename, compute word counts.

**Stage 3: Romantic passage extraction** (block 7). Define a 60-term romance vocabulary and 23 compiled regex phrase patterns. Split each novel into paragraphs, score every paragraph by keyword hits + phrase pattern matches + dialogue bonus, detect themes from a 7-category taxonomy, compute sentiment intensity from positive/negative word counts. Process all 419 novels. Result: 58,241 candidate passages.

**Stage 4: Refinement and false-positive reduction** (block 8). Tighten the lexicon (remove overly broad terms like "heart" and "court"), add front-matter trimming (skip to first CHAPTER/BOOK/LETTER marker), require either a strong pattern match OR multiple distinct theme signals. Passages reduced from 58,241 to 44,560.

**Stage 5: Aggregation and analysis** (blocks 9-11). Compute per-decade statistics (mean/weighted frequency, intensity, theme counts), calculate decade-over-decade deltas for inflection detection, rank top passages by intensity and score with quality filters (remove tables of contents, collected works, misdated anthologies), build early-vs-late comparison, assemble final JSON.

The key difference from the 50-novel run: **zero LLM calls for classification**. The model decided the 419-novel scale warranted a purely computational approach -- heuristic scoring with iterative refinement rather than batched LLM judgment. It read 309MB of text, processed it entirely in code, and used only 93K tokens for the root LM conversation.

## The Code

### Block 6: Gutenberg Cleaning + Metadata Extraction (68 lines)

The model handled three variants of Gutenberg header format and extracted title/author from metadata fields:

```python
def clean_gutenberg_text(text):
    text = text.replace('\ufeff', '')
    start_markers = [
        "*** START OF THE PROJECT GUTENBERG EBOOK",
        "*** START OF THE PROJECT GUTENBERG",
        "***START OF THE PROJECT GUTENBERG",
    ]
    end_markers = [
        "*** END OF THE PROJECT GUTENBERG EBOOK",
        "*** END OF THE PROJECT GUTENBERG",
        "***END OF THE PROJECT GUTENBERG",
    ]
    start_idx = 0
    for m in start_markers:
        i = text.find(m)
        if i != -1:
            nl = text.find("\n", i)
            start_idx = nl + 1 if nl != -1 else i
            break
    text2 = text[start_idx:]
    end_idx = len(text2)
    for m in end_markers:
        i = text2.find(m)
        if i != -1:
            end_idx = min(end_idx, i)
    text2 = text2[:end_idx]
    return text2.strip()

def extract_title_author(raw_text, filename):
    title = None
    author = None
    m = re.search(r'^\s*Title:\s*(.+)$', raw_text, flags=re.I|re.M)
    if m:
        title = m.group(1).strip()
    m = re.search(r'^\s*Author:\s*(.+)$', raw_text, flags=re.I|re.M)
    if m:
        author = m.group(1).strip()
    if not title:
        title = re.sub(r'\.txt$', '', filename).replace('_', ' ')
    if not author:
        author = "Unknown"
    return title, author
```

**Output:** 419 novels indexed with title, author, word count, and character count. Total: 54M words.

The scoring function combined multiple signals:

```python
def romance_score(par):
    low = par.lower()
    words = re.findall(r"\b[\w'-]+\b", low)
    word_hits = sum(1 for w in words if w in romance_terms)
    pattern_hits = sum(1 for pat in compiled_patterns if pat.search(low))
    quote_bonus = 1 if '"' in par or '\u201c' in par else 0
    return word_hits + 2*pattern_hits + quote_bonus

def sentiment_intensity(par):
    low = par.lower()
    pos = sum(len(re.findall(rf"\b{re.escape(w)}\b", low)) for w in positive_words)
    neg = sum(len(re.findall(rf"\b{re.escape(w)}\b", low)) for w in negative_words)
    exclam = par.count("!")
    raw = pos + neg + 0.5*exclam
    if raw <= 1: return 1
    elif raw <= 3: return 2
    elif raw <= 6: return 3
    elif raw <= 10: return 4
    else: return 5
```

**Output:** 58,241 romantic passages extracted from 419 novels.

### Block 8: False Positive Reduction (94 lines)

The model refined its own extraction, tightening the lexicon and adding front-matter detection:

```python
def trim_frontmatter(clean):
    markers = [
        r"\nCHAPTER\s+I\b", r"\nCHAPTER I\b", r"\nBOOK I\b", r"\nLETTER I\b",
        r"\nPART I\b", r"\nI\.\n", r"\nCHAP\. I\b"
    ]
    candidates = []
    for pat in markers:
        m = re.search(pat, clean, flags=re.I)
        if m:
            candidates.append(m.start())
    if candidates:
        idx = min(candidates)
        if idx < len(clean) * 0.35:
            return clean[idx:].strip()
    return clean

def refined_score(par):
    low = par.lower()
    words = re.findall(r"\b[\w'-]+\b", low)
    word_hits = sum(1 for w in words if w in strong_terms)
    pattern_hits = sum(1 for pat in strong_compiled if pat.search(low))
    themes = detect_themes(par)
    return word_hits + 2*pattern_hits, pattern_hits, themes
```

Passages accepted only if: `pattern_hits >= 1` OR `score >= 4` OR `(score >= 3 and len(themes) >= 2)`. This reduced false positives from front matter, tables of contents, and generic "heart/court" mentions.

**Output:** 58,241 → 44,560 passages (23% reduction in noise).

def bad_top_quote(q):
    low = q.lower()
    if len(re.findall(r"\b\d+--", q)) >= 3: return True
    if "contents" in low or "illustrations" in low: return True
    if q.count(";") > 8 and q.count(".") < 5: return True

bad_title_markers = [

### Inflection Points

**1810s to 1820s -- The adventure dip.** Weighted frequency drops from 9.32 to 5.10 (-4.22), the century's largest decline. The 1810s include Austen (4 novels), Hoffmann, and The Monk -- romance-dense writing. The 1820s shift to Cooper (5 novels: Last of the Mohicans, The Spy, The Deerslayer, The Pioneers, The Prairie) -- frontier adventure fiction where romance is incidental. Intensity also drops from 1.78 to 1.60.

**1820s to 1830s -- Romantic resurgence.** Frequency rebounds from 5.10 to 8.79 (+3.70), the century's largest increase. The 1830s bring Eugene Onegin, Henrietta Temple, Mathilda, Notre-Dame de Paris, and A Hero of Our Time -- Continental Romanticism at its peak, where love and suffering are inseparable. Physical affection enters the dominant themes for the first time.

**1800s to 1810s -- Highest intensity spike.** Mean intensity jumps from 1.46 to 1.78 (+0.32), the largest sentiment increase. The 1810s corpus concentrates romance-heavy authors (Austen, Hoffmann, Matthew Lewis) whose passages carry denser emotional language.

**1830s to 1840s -- Intensity softens.** Intensity drops from 1.78 to 1.59 (-0.18) despite frequency holding near 8.7. The 1840s broaden the sample with social novels (Dickens, Thackeray, Gaskell) where romance is woven into larger plots rather than being the primary emotional register. Romantic suffering displaces proposal/marriage as a dominant theme.

**1880s to 1890s -- End-of-century decline.** Frequency drops from 8.75 to 7.10 (-1.65) and intensity falls to 1.45, the century low. The 1890s corpus includes genre diversification: detective fiction (Doyle, 6 novels), science fiction (Wells), horror (Stoker), and social realism (Chopin, Norris). Romance shares the stage with other concerns.

## Emergent Behaviors

### What the model invented (unprompted)

1. **Purely computational pipeline.** Unlike the 50-novel run which used `llm_query()` for classification, the 500-novel model decided the scale warranted a zero-LLM approach. It built heuristic extraction, scoring, theme detection, and sentiment estimation entirely in code. This is a strategic decision about cost and reliability at scale.

2. **Two-pass extraction with self-refinement.** Block 7 produced 58,241 passages; the model then wrote block 8 to tighten its own criteria and remove false positives, reducing to 44,560. It refined its own work without being asked.

3. **Front-matter trimming heuristic.** Seven different chapter/book/letter markers, with a 35% position guard (`idx < len(clean) * 0.35`) to avoid trimming mid-book text that happens to start a new section.

4. **Controlled theme taxonomy.** Seven categories with multi-word phrase matching. Same design pattern as the 50-novel run -- the model independently re-derived the need for enumerable labels when building a pipeline where outputs feed into aggregation code.

5. **Multi-signal passage scoring.** Three orthogonal signals: keyword density (bag of words), phrase pattern matching (compiled regex), and dialogue detection (quotation mark bonus). The refinement pass added a fourth: theme diversity (requiring multiple distinct themes for borderline passages).

6. **Quality-aware passage ranking.** Three layers of filtering on top passages: (a) sort by intensity + score + length, (b) remove structural artifacts (tables of contents, numbered lists), (c) remove collected works and anthologies by title pattern matching. The model recognized that its highest-scoring "passage" was actually a catalog of romance novel titles.

7. **Dual frequency metrics.** Both per-novel mean and corpus-weighted frequency. These can diverge when novel lengths vary widely -- a 500K-word novel with moderate frequency weighs more in the weighted metric than a 30K-word novel with high frequency. The model computed both without being asked.

8. **Early-vs-late cross-validation.** Compared the 71-novel early group against the 150-novel late group, noting that the early group's volatility is explained by the smaller and less diverse sample.

### What the model chose NOT to do

1. **No LLM calls.** Zero `llm_query()` or `rlm_query()` calls. The 50-novel run made 5 batched `llm_query` calls for classification; the 500-novel run solved the entire problem in code. The model made a cost-efficiency tradeoff: heuristic classification at 419-novel scale vs. LLM classification at 50-novel scale.

2. **No recursion.** Despite having `rlm_query()` available, the model handled everything at the root level. A recursive approach (decade coordinators → per-novel analyzers) was available but unnecessary.

3. **No novel-level chunking.** The model read entire novels into memory and split by paragraph. At 309MB this is feasible because the processing is purely computational -- no context window to manage.

## How RLM Differs From Claude Code

This experiment reinforces the architectural difference surfaced in the 50-novel analysis.

### Claude Code: LLM orchestrates tools

```
LLM thinks -> calls tool -> sees result -> LLM thinks -> calls tool -> ...
```

Every step passes through the LLM. To process 419 novels, Claude Code would either write one monolithic Bash script and hope it works, or iterate through hundreds of agent loop turns -- each one a full LLM call with growing conversation context. The LLM IS the program; tools are its I/O.

### RLM: Code orchestrates LLM

```python
for novel in novels:           # code loop, runs at CPU speed
    passages = extract(novel)  # code -- regex, no LLM needed
    scored = score(passages)   # code -- heuristic, no LLM needed
    results.append(aggregate(scored))  # code
```

The for loop is in code, not in the agent loop. In this run, the LLM was never called as a subroutine at all -- the model decided everything could be handled computationally. The 11 iterations of the root LM conversation were used to develop and refine the pipeline, not to execute it.

### The scaling insight

The 50-novel run used LLM calls for classification (heuristic extraction → LLM judgment → code aggregation). The 500-novel run dropped the LLM entirely (heuristic extraction → heuristic scoring → code aggregation). **The model adapted its strategy to the scale of the problem.**

This is not something a fixed tool-using agent can do. Claude Code's tools don't change based on problem size. But RLM's model writes its own tools -- and at 419 novels, it wrote tools that don't need the LLM at all. The ability to make this cost/quality tradeoff autonomously is a capability unique to the code-orchestrates-LLM paradigm.
